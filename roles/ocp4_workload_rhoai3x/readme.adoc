= Red Hat OpenShift AI 3.x Demo — ocp4_workload_rhoai3x

== Overview

AgnosticDv2 workload role that provisions a complete *Red Hat OpenShift AI (RHOAI) 3.3* demo environment on OpenShift 4 (AWS). When provisioning completes, you have a GPU-backed cluster running a live LLM that users can interact with through the RHOAI dashboard's Gen AI Playground, track experiments with MLflow, and build AI pipelines against — all without any manual configuration.

This is an *opinionated* installation designed for the Red Hat Demo Platform. It makes deliberate choices about which RHOAI components to enable, which GPU instance type to use, and which model to serve — all tuned for a single-GPU demo environment. It deploys *one model end-to-end* out of the box (`gpt-oss-20b`). See <<changing-the-model>> to deploy a different model instead.

=== What Gets Deployed

[cols="1,2"]
|===
|Component |Details

|GPU infrastructure
|AWS GPU MachineSet (`g6e.4xlarge`, 1x NVIDIA L40S, 200 GB root volume)

|NVIDIA GPU Operator
|OLM, `stable` channel — driver managed automatically by OCP Driver Toolkit

|Red Hat OpenShift AI
|OLM, `stable-3.x` channel — Dashboard, Workbenches, KServe, AI Pipelines, LlamaStack, MLflow, Model Registry

|Model serving
|`gpt-oss-20b` via KServe InferenceService (RawDeployment mode, no Service Mesh required)

|Gen AI Playground
|LlamaStackDistribution wired to the deployed model — accessible from the RHOAI dashboard

|AI Pipelines
|DataSciencePipelineApplication in `rhoai-shared`, backed by MinIO

|MLflow
|MLflow instance for experiment tracking, backed by MinIO artifacts
|===

== Requirements

* OpenShift 4.16+ cluster on AWS (the cluster itself is not provisioned by this role)
* The following prerequisite workloads must run before this role:

[cols="1,2"]
|===
|Workload |Purpose

|`ocp4_workload_authentication_htpasswd`
|Creates admin and student user accounts

|`ocp4_workload_nfd`
|Installs the Node Feature Discovery operator, which labels GPU nodes automatically

|`ocp4_workload_minio`
|Deploys in-cluster S3 storage used by AI Pipelines and MLflow for artifact storage
|===

== Usage

This role is deployed using https://github.com/agnosticd/agnosticd-v2[AgnosticDv2], a framework for provisioning cloud-based demo and workshop environments. AgnosticDv2 handles cluster connectivity, variable injection, and workload lifecycle (`provision` / `destroy`). See the https://github.com/agnosticd/agnosticd-v2/blob/main/docs/setup.adoc[AgnosticDv2 setup guide] to install the `agd` CLI and configure credentials before running this role.

=== Base environment

This role has been tested against the *AWS with OpenShift Open Environment* on the https://catalog.demo.redhat.com/catalog?item=babylon-catalog-prod/sandboxes-gpte.sandbox-ocp.prod[Red Hat Demo Platform]. It will need adaptation (instance type, storage class, cluster credentials) if deployed against a different base environment.

=== Sample vars file

Create a vars file (e.g. `agnosticd-v2-vars/openshift-workloads-rhoai.yml`). The `openshift-workloads` config deploys the workloads listed under `workloads:` in sequence to the target cluster.

[source,yaml]
----
tag: main
cloud_provider: none
config: openshift-workloads

requirements_content:
  collections:
  - name: https://github.com/agnosticd/core_workloads.git
    type: git
    version: "{{ tag }}"
  - name: https://github.com/agnosticd/showroom.git
    type: git
    version: v1.5.7
  - name: https://github.com/redhat-telco-adoption/agnosticd-workloads.git
    type: git
    version: "{{ tag }}"

clusters:
- default:
    api_url: "{{ cluster1.sandbox_openshift_api_url }}"
    api_token: "{{ cluster1.sandbox_openshift_api_token }}"

workloads:
- agnosticd.core_workloads.ocp4_workload_authentication_htpasswd
- agnosticd.core_workloads.ocp4_workload_nfd
- agnosticd.core_workloads.ocp4_workload_minio
- redhat_telco_adoption.agnosticd.ocp4_workload_rhoai3x

remove_workloads:
- redhat_telco_adoption.agnosticd.ocp4_workload_rhoai3x
- agnosticd.core_workloads.ocp4_workload_minio
- agnosticd.core_workloads.ocp4_workload_nfd
- agnosticd.core_workloads.ocp4_workload_authentication_htpasswd

# -------------------------------------------------------------------
# ocp4_workload_authentication_htpasswd
# -------------------------------------------------------------------
ocp4_workload_authentication_htpasswd_admin_user: admin
ocp4_workload_authentication_htpasswd_admin_password: "YOUR_ADMIN_PASSWORD"
ocp4_workload_authentication_htpasswd_user_count: 5
ocp4_workload_authentication_htpasswd_user_password: "openshift"

# -------------------------------------------------------------------
# ocp4_workload_minio
# -------------------------------------------------------------------
ocp4_workload_minio_project: minio
ocp4_workload_minio_storage_size: "200Gi"
ocp4_workload_minio_root_user: "minio"
ocp4_workload_minio_root_password: "YOUR_MINIO_PASSWORD"
ocp4_workload_minio_bucket_name:
- rhoai-models
- rhoai-pipelines

# -------------------------------------------------------------------
# ocp4_workload_rhoai3x
# -------------------------------------------------------------------

ocp4_workload_rhoai3x_model_storage_uri: "oci://registry.redhat.io/rhelai1/modelcar-gpt-oss-20b:1.5"
ocp4_workload_rhoai3x_model_serving_runtime_image: "registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2-1774351144"
----

=== Provision

[source,bash]
----
./bin/agd provision --guid <your-guid> --config openshift-workloads-rhoai --account <your-account>
----

=== Destroy

[source,bash]
----
./bin/agd destroy --guid <your-guid> --config openshift-workloads-rhoai --account <your-account>
----

=== Provision timing

Total provision time is approximately *60-90 minutes*, dominated by GPU node provisioning and model image pulls.

[cols="2,1"]
|===
|Phase |Approximate duration

|GPU node provisioning (MachineSet scale-up)
|~10 min

|NVIDIA GPU Operator + ClusterPolicy ready
|~15 min

|RHOAI Operator + DataScienceCluster ready
|~10 min

|MLflow + AI Pipelines setup
|~3 min

|Model image pull (`gpt-oss-20b` ~41 GB + vLLM ~15 GB)
|~20 min

|Model load + InferenceService ready
|~5 min

|Gen AI Playground (LlamaStackDistribution) ready
|~5 min
|===

== Changing the Deployed Model

The role deploys a single model end-to-end. To use a different model, override these variables in your vars file:

[source,yaml]
----
# Model identity — used as the InferenceService name, ServingRuntime name, and display name
ocp4_workload_rhoai3x_model_name: "my-model"
ocp4_workload_rhoai3x_model_display_name: "My Model"

# Where to pull the model from
# OCI ModelCar (Red Hat registry):
ocp4_workload_rhoai3x_model_storage_uri: "oci://registry.redhat.io/rhelai1/modelcar-<model>:<tag>"
# S3 bucket:
# ocp4_workload_rhoai3x_model_storage_uri: "s3://<bucket>/<path>"
# HuggingFace:
# ocp4_workload_rhoai3x_model_storage_uri: "hf://<org>/<repo>"

# vLLM container image — see image selection table below
ocp4_workload_rhoai3x_model_serving_runtime_image: "quay.io/modh/vllm:rhoai-2.25-cuda"

# Optional: extra vLLM CLI args
# ocp4_workload_rhoai3x_model_extra_args: ["--max-model-len=8192"]

# GPU and compute sizing — adjust to fit your model
ocp4_workload_rhoai3x_model_gpu_count: "1"
ocp4_workload_rhoai3x_model_memory_requests: "4Gi"
ocp4_workload_rhoai3x_model_memory_limits: "8Gi"
----

=== Finding a model

https://huggingface.co/RedHatAI[RedHatAI on Hugging Face] is the best starting point for Red Hat validated models. Each model card documents the quantization format, context length, and hardware requirements — check those before selecting an image (see below).

=== Image selection

Choose the vLLM image based on your model's quantization format, which is listed in the model card or in the model's `config.json` (`quantization_config.quant_type`).

[cols="1,2,2"]
|===
|Quantization |Image |Notes

|mxfp4 (e.g. `gpt-oss-20b:1.x`)
|`registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2-1774351144`
|Required — generic images do not support mxfp4. Do *not* pass `--quantization`; the image auto-detects it.

|fp16 / bf16 (standard models)
|`quay.io/modh/vllm:rhoai-2.25-cuda`
|Works with most public HuggingFace models.

|GPTQ / AWQ
|`quay.io/modh/vllm:rhoai-2.25-cuda`
|Pass `--quantization gptq` or `--quantization awq` via `model_extra_args`.
|===

=== GPU sizing

The default `g6e.4xlarge` has 1x NVIDIA L40S (48 GB VRAM). For larger models:

[cols="1,1,1,2"]
|===
|Instance type |GPUs |VRAM |Suitable for

|`g6e.4xlarge`
|1x L40S
|48 GB
|Up to ~20B parameters (mxfp4/fp4), ~7B in fp16

|`g6e.12xlarge`
|4x L40S
|192 GB
|Up to 70B in fp16 — use `--tensor-parallel-size=4`

|`g6e.48xlarge`
|8x L40S
|384 GB
|Up to 405B in fp16 — use `--tensor-parallel-size=8`
|===

For multi-GPU deployments, also set:

[source,yaml]
----
ocp4_workload_rhoai3x_gpu_instance_type: "g6e.12xlarge"
ocp4_workload_rhoai3x_model_gpu_count: "4"
ocp4_workload_rhoai3x_model_extra_args: ["--tensor-parallel-size=4"]
----

== MLflow instrumentation

To track experiments from a workload running inside the cluster:

[source,bash]
----
export MLFLOW_TRACKING_TOKEN="$(oc whoami --show-token)"
export MLFLOW_TRACKING_URI="https://data-science-gateway.apps.<cluster-domain>/mlflow"
export MLFLOW_WORKSPACE="<your-namespace>"
----

NOTE: The token is session-scoped. Refresh it with `oc whoami --show-token` each time.

== Verification

[source,bash]
----
# GPU node — Ready and nvidia.com/gpu allocatable
oc get nodes -l nvidia.com/gpu.present=true
oc get node <gpu-node> -o jsonpath='{.status.allocatable.nvidia\.com/gpu}'

# NVIDIA GPU Operator
oc get csv -n nvidia-gpu-operator | grep gpu-operator
oc get clusterpolicy gpu-cluster-policy -o jsonpath='{.status.state}'

# RHOAI Operator
oc get csv -n redhat-ods-operator | grep rhods

# DataScienceCluster component status
oc get datasciencecluster default-dsc -o wide

# Model serving
oc get inferenceservice -n rhoai-shared
oc get servingruntime -n rhoai-shared

# Gen AI Playground
oc get llamastackdistribution -n rhoai-shared

# MLflow
oc get mlflow -n redhat-ods-applications

# AI Pipelines
oc get datasciencepipelinesapplication -n rhoai-shared
----
